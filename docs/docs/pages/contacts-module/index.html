iOS 原生模块 / 系统服务

contacts

联系人读取、编辑、选择器和 vCard 导入导出。

contacts 访问系统通讯录，支持权限、读取、搜索、联系人编辑、分组、系统选择器、vCard 导入导出和变更历史。联系人属于敏感数据，默认只读取当前功能需要的字段和数量。

#预期效果

示例会展示联系人选择、读取、vCard 导入导出和取消状态处理。

#什么时候用

  • 让用户选择一个或多个联系人，并在 App 内继续处理。
  • 搜索联系人、展示联系人卡片、读取手机号或邮箱。
  • 创建、编辑、删除联系人或分组，并用 save() 提交事务。
  • 导入导出 vCard，或读取变更历史做增量同步。

#核心模型

读取前先检查 is_authorized()；没有权限时再调用 request_access()。联系人修改不是立即落盘：add_person、remove_person、edit_person、分组成员修改会进入待提交状态，成功时调用 save()，失败时调用 revert()。

系统选择器 pick_contact() / pick_contacts() 适合让用户主动选择数据，比无提示读取全部联系人更稳。列表页读取联系人时使用 limit 和 offset，避免一次性加载图片或 vCard。

#使用规则

  • 读取前先判断 is_authorized()；没有权限时再调用 request_access()，不要无提示批量读取。
  • 列表页必须使用 limit 和 offset，不要一次性加载全部联系人图片。
  • 修改联系人、分组或成员后用 save() 提交；失败时用 revert() 回滚。
  • include_image_data 和 include_vcard 会增加内存和隐私暴露，只在确实需要时打开。
  • iOS 限制通讯录权限时，用 manage_limited_access() 让用户调整可访问联系人。

#失败路径

情况应该怎么处理
用户拒绝通讯录权限显示未授权状态，不继续读取。
pick_contact() 返回空视为用户取消，保持当前数据。
保存联系人失败调用 revert() 回滚待提交修改。
限制通讯录访问提供 manage_limited_access() 入口。

#常见错误

错误写法后果修正
首屏直接读取所有联系人隐私体验差且可能很慢由按钮触发，并使用 limit。
修改后忘记 save()变更没有提交成功路径调用 save()。
保存失败后继续使用待提交对象状态和系统通讯录不一致捕获异常后 revert()。
默认加载 include_image_data=True内存和隐私成本高只在头像页面按需加载。

后台任务、保活与 BGTask 调度。

后台任务、保活与 BGTask 调度。

#预期效果

示例会展示后台任务开始、剩余时间读取和结束清理。

#适用场景

后台任务、保活、应用状态和 BGTask 调度。它只能延长或调度有限后台执行，不是无限后台运行能力。

#核心模型

begin_task() 只能争取一小段后台收尾时间，适合保存状态、完成短请求或清理资源。它不是长期后台运行许可。调用后必须在 finally 里 end_task()。

schedule_refresh(...) 和 schedule_processing(...) 只是向系统提交后台任务请求，实际触发时间由 iOS 决定，不能承诺精准时间。earliest_date 使用 Unix timestamp，0 表示让系统使用默认最早时间。

#API 参考

类型API签名说明
functionbegin_taskbegin_task() -> Any请求有限后台执行时间，成功返回真值。
functionend_taskend_task() -> Any提前结束后台任务。
functionremaining_timeremaining_time() -> Any剩余后台时间，单位秒。
functionapp_stateapp_state() -> Any返回 active、inactive、background 或 unknown。
functionstart_keep_alivestart_keep_alive() -> Any启用后台音频会话以延长活跃时间。
functionstop_keep_alivestop_keep_alive() -> None关闭 keep-alive 音频会话，释放后台保活资源。
functionschedule_refreshschedule_refresh(identifier, earliest_date=0) -> Any调度 BGAppRefreshTask。
functionschedule_processingschedule_processing(identifier, earliest_date=0, requires_network=False, requires_charging=False) -> Any调度 BGProcessingTask。

#使用规则

  • begin_task 后必须用 finally 调用 end_task。
  • 后台时间有限，不要依赖它执行无限任务。
  • remaining_time() 只能作为参考，系统随时可能收紧后台时间。
  • start_keep_alive() 会占用音频会话，只有确实需要长时间保持活跃时才使用。
  • schedule_refresh / schedule_processing 需要稳定 identifier，并且实际触发时间由系统决定。

#常见误区

误区正确做法
以为 begin_task() 可以无限运行只做收尾、保存、短下载等有限任务
忘记结束后台任务用 try/finally 包住工作
依赖 BGTask 准时执行设计成“系统有机会时刷新”，不要承诺精确时间
写 requires_external_power参数不存在，使用 requires_charging

#失败路径

情况应该怎么处理
begin_task() 返回假值缩短任务，提示无法继续后台执行。
remaining_time() 很短立即保存状态并退出后台工作。
调度返回 False检查 identifier 是否配置，显示调度失败。
用户需要完成提醒配合 notification，不要依赖后台准时运行。
